## Critical Gap: Webhook Metadata Reconstruction

### The Problem

Phase 1 (Plan 01-02) optimized Stripe metadata to use shortened keys to stay under the 500-char limit:

```typescript
// Checkout API creates metadata (src/app/api/checkout/route.ts:107):
items: JSON.stringify(items.map(i => ({ pid: i.productId, vid: i.variantId, qty: i.quantity })))
```

Phase 2 (Plan 02-01) created a session-details API that **correctly** reconstructs full item data from these shortened keys:

```typescript
// Session-details API (src/app/api/checkout/session-details/route.ts):
const product = getProductById(shortItem.pid);
items.push({ name: product.name, quantity: shortItem.qty, price: product.salePrice || product.price });
```

But the webhook handler (modified in Phase 2 Plan 02-02 for email idempotency) was **NOT updated** to handle the new metadata format:

```typescript
// Webhook (src/app/api/webhooks/stripe/route.ts:406):
items = JSON.parse(metadata.items);  // Gets [{ pid, vid, qty }] — NOT [{ name, quantity, price }]
```

### Impact

| Component | Impact |
|-----------|--------|
| Order persistence | Stores `{pid, vid, qty}` instead of `{name, quantity, price}` — corrupt data |
| Customer email | Renders `item.name` as undefined — malformed email |
| Store notification | Same — malformed notification |
| Success pages | **NOT affected** — uses session-details API which reconstructs correctly |
| Admin panel | Shows corrupt order data |

### Fix Required

The webhook handler at line 403-413 must reconstruct full item data from shortened keys, using the same pattern as `session-details/route.ts`:

**File:** `src/app/api/webhooks/stripe/route.ts` (line 403-413)

Replace direct JSON.parse with reconstruction logic:
1. Parse shortened items `[{ pid, vid, qty }]`
2. For each item, call `getProductById(pid)` to get full product data
3. Build proper `OrderItem` objects `{ name, quantity, price, productType }`
